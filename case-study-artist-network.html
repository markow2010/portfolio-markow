Case Study · 2023

Artist Social Network Visualization

An interactive web application that reveals the hidden structure of artist collaboration networks using graph theory and real-time visualization.

  • RoleFull-Stack Developer
  • StackPython · Flask · Bokeh
  • Architecture3-Tier (GUI / App / Data)
  • Year2023
View Source Companion App

The Problem

Music collaboration is a rich, graph-shaped dataset — artists feature on each other's tracks, form implicit communities, and exhibit surprising "centrality" patterns that reveal who the genre's connectors and gatekeepers really are. But most listeners only see a flat list of songs. There was no tool that let you explore these relationships interactively.

The Approach

I designed and built a Flask web application with a clean 3-tier architecture so each layer could be developed, tested, and scaled independently:

Interface Layer
gui/

HTML templates + Bootstrap UI. Handles user input, artist selection, and renders the interactive Bokeh network graph in the browser.

↓
Application Layer
applayer/

Core logic: graph expansion, centrality computation, and request handling. Converts raw data into visualized networks.

↓
Data Layer
datalayer/

Artist data retrieval and relationship modeling. Isolated so data sources can be swapped without touching business logic.

Key Features

Interactive Network Expansion

Click any artist node to expand their collaboration neighborhood. The graph grows dynamically as you explore.

Centrality Measures

Computes degree, betweenness, and closeness centrality — revealing the network's real power brokers, not just the most famous names.

Real-Time Rendering

Powered by Bokeh for browser-native, GPU-accelerated graph plotting — no image refresh, no flicker.

Clean Separation of Concerns

The 3-tier design means the data source could be swapped from CSV to SQL to a streaming API without touching the UI.

Implementation Snapshot

The application layer exposes a clean API to the Flask routes. A simplified view: 

from flask import Flask, render_template, request
from applayer import network, centrality

app = Flask(__name__)

@app.route("/")
def index():
    return render_template("index.html")

@app.route("/expand", methods=["POST"])
def expand():
    artist = request.json["artist"]
    graph  = network.expand(artist, depth=2)
    stats  = centrality.compute(graph)
    return {"graph": graph.to_json(), "stats": stats}

What I Learned

  • Architecture matters early. The 3-tier split felt like overkill at first, but when I later forked the project into a companion app, reusing the applayer/ and datalayer/ was trivial.
  • Graph theory in the wild. Centrality isn't abstract — it visually changes who looks "important" in a network.
  • Bokeh over matplotlib. For interactive visualization in the browser, Bokeh is dramatically better. Matplotlib is for reports; Bokeh is for apps.
  • Flask is perfect for prototypes. I had a working app in an afternoon. No build step, no config churn.
